# AI Chatbot MVP 方案说明

## 1. 你对需求的理解

题目要求构建一个**面向内部员工的 AI Chatbot MVP**，核心是在网页中实现"多轮对话 + 工具调用"的完整闭环。三个关键信息：

- **AI**：不强制接入真实 LLM，但必须有"智能感"——能理解用户意图并分流到不同工具。
- **工具**：至少一个工具调用场景，且由系统自动判断触发（不是按钮点击），结果回填到对话中。
- **MVP**：跑通核心流程即可，明确边界，不追求完整产品量级。

更深层的考核意图：看候选人能否用 AI 工具高效完成"需求拆解 → 架构设计 → 编码实现 → 调试验证 → 方案复盘"全过程，以及是否具备**工程判断力**——不是 AI 说什么就做什么。

---

## 2. 你如何拆解 MVP 范围

按三层架构纵向拆解，每层拆成最小可验证单元：

```
第一层：前端（用户可见的交互层）
  ├── 聊天页面（消息列表 + 输入框）
  ├── 消息气泡（用户消息 + AI 回复 + 重试按钮）
  ├── 工具调用卡片（参数/结果可折叠查看）
  └── 多标签页（新建/切换/关闭/编号回收）

第二层：后端（请求处理与意图路由）
  ├── POST /api/chat 接口
  ├── 会话管理（conversationId 贯穿 + 内存消息数组）
  └── Mock LLM Provider（意图识别 → 工具路由 → 回复生成）

第三层：工具（可扩展的业务能力）
  ├── 查询类：时间查询、员工信息查询
  ├── 计算类：数学表达式求值
  └── 管理类：待办 CRUD（创建/查看/完成/删除）
```

**开发顺序**：后端接口骨架 → Mock Provider（先跑通闲聊兜底） → 逐个工具接入 → 前端对接 → 多标签页 → 智能度打磨。

## 3. 第一版做了什么，明确不做什么

### 做了

| 模块 | 内容 | 关键文件 |
|---|---|---|
| 前端 | React + Vite + Tailwind，消息气泡（含重试）、工具调用卡片（可折叠）、多标签页 | `client/src/App.tsx`、`ChatContainer.tsx` |
| 后端 | Express + tsx 热加载，`POST /api/chat` JSON 响应，会话管理 | `server/src/index.ts`、`routes/chat.ts` |
| Mock LLM | 正则意图识别（7 种意图 → 7 个工具）、3 层守卫防误判、4 层智能回复 | `intentDetector.ts`、`responseBuilder.ts` |
| 工具 | 时间查询、计算器（含日期守卫）、待办 CRUD（模糊匹配）、员工查询（三级置信） | `server/src/tools/*.ts` |
| 会话 | conversationId 贯穿多轮对话，服务端内存 Map 存储 | `conversationManager.ts` |
| 错误处理 | 前端 retry 机制、工具失败友好提示、建待办的疑问句/标题有效性/日期格式守卫 | 散见各模块 |

### 明确不做

| 不做的事 | 理由 |
|---|---|
| 真实 LLM 接入 | Mock 模式已满足核心流程验证；切换只需设 `OPENAI_API_KEY`，零代码改动 |
| 用户登录 / 权限 | 内部工具 MVP 阶段，登录不是核心流程 |
| 数据库持久化 | 内存存储足够验证流程；换成 SQLite 只需替换 store 层 |
| 复杂工具编排 / 多工具并行 | MVP 阶段单工具即可展示工具调用闭环 |
| UI 动画 / 精美化 | 题目关注的是工程和 AI 使用深度，不是 UI 竞赛 |

## 4. 你为什么这样取舍

| 取舍 | 为什么 |
|---|---|
| **Mock 优先，不接真实 LLM** | 题目明确要求"没有 API Key 也能体验"。Mock Provider 内部已实现完整意图识别→路由→回复链路，真实 LLM 接入只需替换一个 Provider 类，不影响架构。 |
| **7 个工具而非 1 个** | 1 个工具不够体现路由判断价值。7 个工具覆盖查询/计算/CRUD 三大类，需要真实的意图优先级、参数提取、结果格式化。 |
| **内存存储而非数据库** | 待办数据量极小，零配置零依赖，评审 clone 即跑。 |
| **多标签页** | 评审可能需同时测试多个对话场景，单页面来回清空效率低。标签编号通过 `findLowestAvailable()` 实现自动回收。 |
| **4 层智能回复** | 纯规则引擎也能有"拟真对话感"：精确意图→上下文感知→话题感知→默认兜底，配合随机过渡语轮转，让 Mock 回复不像机械模板。 |
| **不做登录** | 内部员工场景 MVP 阶段，增加 OAuth/JWT 徒增复杂度，对核心流程无增量价值。 |

## 5. 核心流程如何运行

以"给我写一个待办明天下午5点我有一个会议"为例，完整链路 5 步：

```
第 1 步：用户在前端输入
  "给我写一个待办明天下午5点我有一个会议"
              │
第 2 步：前端 POST /api/chat → 后端 chat.ts 路由
              │
第 3 步：MockProvider.processMessage()
  ├── IntentDetector.detect()
  │   ├── 时间查询模式 → 不匹配
  │   ├── 计算器模式   → 不匹配（"5" 无运算符）
  │   ├── 完成/删除模式 → 不匹配
  │   ├── 查看待办模式 → 不匹配
  │   ├── 创建待办模式 → 匹配 "给.*待办" ✅
  │   ├── 疑问句守卫   → 不触发（不是"可以...吗"）
  │   ├── extractTodoTitle() → 提取 "明天下午5点我有一个会议"
  │   ├── extractPriority()  → 默认 "medium"
  │   └── extractDate()      → "明天" → "2026-06-29"
  │
  ├── MockProvider.routeIntent()
  │   └── createTodoTool.execute({ title, priority, dueDate })
  │       └── todoStore.push(todoItem)
  │
  └── ResponseBuilder.buildFinalResponse()
      └── "搞定了，已创建待办..." + 截止日期倒计时提醒
              │
第 4 步：后端返回 JSON
  { conversationId, message: { content: "..." },
    toolCalls: [{ name: "createTodo", arguments: {...}, result: {...} }] }
              │
第 5 步：前端 React 渲染
  ├── MessageBubble 组件 → 显示自然语言回复
  └── ToolCallCard 组件 → 可折叠工具调用详情（使用了什么参数、返回了什么结果）
```

## 7. 多轮上下文如何处理

**服务端维护 messages 数组 + conversationId 贯穿**：

```
第 1 轮：
  前端发 → { message: "我叫张三，我在技术部", conversationId: null }
  服务端创建 conv-1，存入 [system, user]
  后端返回 → { conversationId: "conv-1", ... }

第 2 轮：
  前端发 → { message: "查一下我部门的同事", conversationId: "conv-1" }
  服务端从 ConversationManager 取出 conv-1 的完整历史
  MockProvider 看到完整 messages 上下文
  意图识别仍靠当前消息的正则匹配（Mock 局限）
  后端返回工具调用 + 回复
```

**设计要点**：
- 前端通过 `conversationId` 将每次请求关联到同一会话
- 后端 `ConversationManager` 用内存 Map 维护所有会话的消息数组
- 服务重启后历史丢失（MVP 可接受范围）
- 前端多标签页各自维护独立 `conversationId`，标签间互不干扰

**Mock 模式的上下文局限**：意图识别靠正则匹配当前输入，无法真正"理解"上文（如"用高优先级"无法自动关联上一个待办）。这是**有意为之**的限制——证明我清楚 Mock 不是真 LLM，并预留了 `messages` 全量传给 Provider 的接口，后续接入真实 LLM 后上下文理解能力自然提升。

## 8. 工具调用如何判断、执行和返回结果

### 判断：正则意图识别 + 优先级路由 + 多层守卫

按优先级顺序依次匹配正则，**先命中先返回**：

| 优先级 | 意图 | 触发正则示例 | 关键守卫 |
|---|---|---|---|
| 1 | 时间查询 | `现在几点` `看看时间` `当前日期` | — |
| 2 | 计算器 | `128*256` `计算一下` `等于多少` | 日期守卫：排除 `2026-07-01` |
| 3 | 完成待办 | `把周报完成了` `周报做完了` `搞定代码` | 标题长度守卫：`完成了` 口语不触发 |
| 4 | 删除待办 | `把周报删了` `删掉测试` `去掉会议` | 标题长度守卫 |
| 5 | 创建待办 | `给我写个待办` `帮我建一个` `记一下` | 疑问句守卫 + 标题有效性守卫 |
| 6 | 查看待办 | `我有什么待办` `看看待办` `还剩什么待办` | — |
| 7 | 员工查询 | `我部门有哪些人` `谁在技术部` `帮我查张三`（三级置信） | 低置信模式需有人名/部门 |
| 兜底 | 闲聊 | 4 层智能回复 | 精确→上下文→话题→默认 |

### 执行：Typed Tool Registry

```typescript
// 每个工具都实现统一的 ToolDefinition 接口
const tool = toolRegistry.get(intent.type);
const result = await tool.execute(extractedParams);
```

每个工具有独立的类型定义和输入校验：
- `createTodo`：校验标题非空、优先级合法、日期格式 YYYY-MM-DD
- `calculator`：在 `isDateExpression()` 守卫通过后才进入求值
- `queryEmployee`：按人名或部门筛选 Mock 数据

### 返回：双通道结果

后端同时返回两条信息：

| 通道 | 内容 | 前端展示 |
|---|---|---|
| `message.content` | 自然语言回复（含过渡语 + 格式化数据 + 主动建议） | `MessageBubble` 组件 |
| `toolCalls[]` | 结构化记录：工具名、参数、返回数据 | `ToolCallCard` 折叠卡片 |

这样评审不仅能看"对话像不像"，还能看"工具调用对不对"。

## 9. Mock 模式如何启用

**零配置默认启用**，这是刻意设计的启动体验：

```bash
cd server && npm run dev
# 输出：[Server] Mock 模式: 启用 (无 API Key)
```

技术实现（`server/src/services/llm/mockProvider.ts`）：

```typescript
const llmProvider = process.env.OPENAI_API_KEY
  ? new OpenAIProvider(process.env.OPENAI_API_KEY)
  : new MockProvider();
```

**MockProvider 内部不调用任何外部 API**，全部本地运算：
- `intentDetector.ts`（228 行）：正则引擎，7 种意图 + 参数提取 + 多层守卫
- `responseBuilder.ts`（374 行）：4 层智能回复 + 随机过渡语轮转 + 上下文感知

**切换到真实 LLM 只需一行环境变量**：

```bash
# 复制模板
cp .env.example .env
# 填入你的 Key
OPENAI_API_KEY=sk-your-key-here
# 重启后端即可，零代码改动
```

## 10. 当前实现的限制和后续优化方向

### 限制

| 类别 | 限制 | 影响 |
|---|---|---|
| 意图识别 | 纯正则匹配，无法理解复杂语义 | "帮我记一下那个事情" 在无"待办"关键词时无法识别 |
| 上下文 | Mock 模式无法真正利用对话历史 | "改成高优先级" 无法关联上一个待办 |
| 存储 | 内存 Map，服务重启数据丢失 | 仅适合开发验证和演示 |
| 工具 | 不支持串行编排 | 创建待办 + 设置优先级需一句话完成 |
| 并发 | 单进程，无连接池 | 仅单用户场景 |

### 优化方向（按优先级排序）

| 优先级 | 方向 | 方案 | 预期收益 |
|---|---|---|---|
| 🔴 高 | 接入真实 LLM | 替换 Provider 为 OpenAI/Gemini，`messages` 数组直接注入 prompt | 意图识别从正则升级为语义理解，覆盖 10x 更多表达方式 |
| 🔴 高 | 数据持久化 | SQLite（单文件零配置）替代内存 Map | 服务重启不丢数据 |
| 🟡 中 | 流式输出 | SSE 替代一次性 JSON | 用户体验接近真实 LLM 对话 |
| 🟡 中 | 消息截断 | 保留最近 20 轮 + 早期对话摘要 | 控制 token 上限，支持更长的多轮对话 |
| 🟢 低 | 多工具编排 | 单次对话允许顺序调用多个工具 | "创建待办然后标记为高优先级" 一步到位 |
| 🟢 低 | 待办提醒 | `setTimeout` 定时检查 + 前端通知 | 真正的"智能提醒"体验 |

## 12. AI 工具输出中，你采纳了什么、修改了什么、拒绝了什么

### 采纳：AI 的设计判断直接正确

| 内容 | 为什么采纳 |
|---|---|
| 整体架构：三层分离（前端/后端/工具） | 职责清晰，每层独立可测 |
| `intentDetector` 从 `mockProvider` 独立成模块 | 468 行拆为 3 文件，各 <250 行，可维护性大幅提升 |
| 待办模糊匹配：`hasWordOverlap()` 算法 | 任意 2 字公共子串匹配，无需 NLP 库，中文友好 |
| 多标签页：`display:none` 保持组件挂载 | 无需 lift state，对话状态天然隔离，切换无闪烁 |
| 工具调用卡片：`isRecord` 类型守卫 | 替代 `typeof === 'object'`（null 也是 object），API 响应更安全 |
| Mock Provider 的 Provider 模式设计 | 符合依赖倒置原则——切换真 LLM 只需注入新 Provider |

### 修改：AI 给的初始方案有 bug 或不够好，我定位后修正

| 内容 | AI 输出的问题 | 我的修改 | 我的判断过程 |
|---|---|---|---|
| 计算器正则 | `[\d+\-*/]{3,}` 直接匹配，把 `2026-07-01` 当成减法 | 加 `isDateExpression()` 守卫函数，计算器前先排除 YYYY-MM-DD 格式 | 创建待办含截止日期时误触计算器，我发现这是正则太过贪婪 |
| 完成/删除正则 | `完成\S+` 匹配了"完成了"（日常口语） | 改为 `完成\S{2,}` + 标题长度 ≥ 2 守卫 | "完成了"是三字但不是指令，是口语，必须过滤 |
| 标签编号 | 模块变量 `tabCounter++` 永不回收 | `findLowestAvailable()` 扫描现有标签取最小可用编号 | 关闭标签 2 后新建应该复用 2，不是跳到 3 |
| 待办 title 提取 | 对"给我写一个待办xxx"未做标题提取 | 加 5 组新的提取正则（含"把XX完成了"倒序结构） | AI 只适配了"创建待办"，没适配口语化表达 |
| 员工查询 | 无置信分级，"帮我查一下"匹配后无参数调用工具返回"未找到" | 三级置信体系：高（人名/部门名）→ 直接触达；中→需匹配；低→需匹配 | "帮我查一下明天的天气"不应触发员工查询 |

### 拒绝：AI 的建议方向正确但不适合当前阶段

| 内容 | 为什么拒绝 |
|---|---|
| AI 建议接入 Pinecone 向量数据库做知识库 | 题目明确"不重点考察复杂知识库"。MVP 阶段不需要。 |
| AI 建议用 WebSocket 流式输出 | Mock 模式下无真实流，徒增复杂度。后续用 SSE 更简单。 |
| AI 建议前端加 Redux/Zustand | 当前仅 `tabs[]` 和 `messages[]` 两个状态，`useState` 足够。过度设计。 |
